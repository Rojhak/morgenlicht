## 2024-05-24 - [Next.js Critical DoS and SSRF Vulnerabilities]
**Vulnerability:** Found multiple vulnerabilities in `next@15.1.9` including Denial of Service via server components, SSRF due to middleware redirect handling, and Cache Key Confusion.
**Learning:** Outdated frameworks, especially in SSR/SSG apps like Next.js, can expose critical infrastructure vulnerabilities (like DoS and SSRF) that aren't specific to application logic but rather framework behavior.
**Prevention:** Regularly audit (`pnpm audit`) and update core framework dependencies (e.g., Next.js) to patched versions, especially when new minor/patch versions are available.
## 2024-04-06 - [Information Exposure & Silent Failures] Fix logging and error handling in Email API
**Vulnerability:** The inquiry API exposed raw internal error objects and stack traces to application logs if `resend.emails.send` crashed, which could be exposed via log management tools. Furthermore, the updated `resend` v6 client uses `{ data, error }` responses instead of throwing exceptions on failure. This meant failures (e.g. invalid API key) would be silently ignored and masked as successful responses because `error` returned by `send()` wasn't checked.
**Learning:** External API clients frequently change their error-handling patterns between major versions (e.g., throwing vs returning an error object). Failing to read documentation during upgrades leads to silent failures. Logging raw `error` objects is an Information Exposure risk because they may contain HTTP request data, internal file paths, or sensitive environmental configuration.
**Prevention:** Explicitly check for error return signatures `const { error } = await sdk.send()`. Sanitize error outputs being logged by using `error instanceof Error ? error.message : 'Unknown error'` to restrict what is serialized into application logs. Use a lazy singleton pattern for heavy HTTP clients to verify configuration safely outside request hot paths.
