## 2024-05-24 - [Next.js Critical DoS and SSRF Vulnerabilities]
**Vulnerability:** Found multiple vulnerabilities in `next@15.1.9` including Denial of Service via server components, SSRF due to middleware redirect handling, and Cache Key Confusion.
**Learning:** Outdated frameworks, especially in SSR/SSG apps like Next.js, can expose critical infrastructure vulnerabilities (like DoS and SSRF) that aren't specific to application logic but rather framework behavior.
**Prevention:** Regularly audit (`pnpm audit`) and update core framework dependencies (e.g., Next.js) to patched versions, especially when new minor/patch versions are available.
## 2024-04-09 - API Payload Type Protection & Secure Email Handling
**Vulnerability:** API routes parsing JSON payloads were vulnerable to runtime TypeErrors when unexpected input types (like arrays or objects) were injected instead of strings, potentially causing crashes or bypassing length checks. Furthermore, the `Resend` SDK was initialized dynamically per-request (which is inefficient in serverless environments and risks throwing on missing env vars) and error logging exposed full error objects (potential information disclosure).
**Learning:** In Next.js API routes, explicit type checking (e.g. `typeof field === 'string'`) must be done before string operations or length validation on parsed request bodies, as malicious clients can spoof any JSON structure. Also, `Resend` v6+ returns `{ error }` rather than throwing, so errors must be explicitly checked, and logging should always be sanitized (`error instanceof Error ? error.message : String(error)`).
**Prevention:** Always implement explicit type guards (`typeof x === 'string'`) in validation functions. Extract heavy SDK initializations to lazy singletons outside handler functions, and always sanitize error objects before server logging to prevent info leakage.
