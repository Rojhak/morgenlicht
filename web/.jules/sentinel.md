## 2024-05-24 - [Next.js Critical DoS and SSRF Vulnerabilities]
**Vulnerability:** Found multiple vulnerabilities in `next@15.1.9` including Denial of Service via server components, SSRF due to middleware redirect handling, and Cache Key Confusion.
**Learning:** Outdated frameworks, especially in SSR/SSG apps like Next.js, can expose critical infrastructure vulnerabilities (like DoS and SSRF) that aren't specific to application logic but rather framework behavior.
**Prevention:** Regularly audit (`pnpm audit`) and update core framework dependencies (e.g., Next.js) to patched versions, especially when new minor/patch versions are available.
## 2026-04-13 - [Strict Input Type Validation and Safe Error Logging]
**Vulnerability:** The sanitization functions (`sanitizeInput`, `sanitizeForSubject`) assumed input was always a string, which could lead to runtime `TypeError` crashes (DoS vectors) if malicious payloads bypassed earlier checks with arrays or objects. Additionally, the API route logged the raw `error` object in the catch block, risking exposure of sensitive stack traces or internal state to server logs.
**Learning:** Even with prior type checking (like in `validateInquiry`), utility functions that perform string operations must implement their own defense-in-depth checks (e.g., `typeof input !== 'string'`) to gracefully handle unexpected types and prevent crashes. Furthermore, error objects should always be safely destructured (e.g., logging only `error.message`) to prevent information leakage.
**Prevention:** Always include explicit type checks in utility functions before calling type-specific methods. When catching exceptions, use `error instanceof Error ? error.message : String(error)` to sanitize the error output before logging or returning it to the client.
